=== PAGE 2 ===
1. システム概要
本アプリは、ユーザーが読書対象の本を登録し、購入予定・購入済み・読書中・読了などの状態管理、評
価、タグ付け、メモ管理を行うための Web アプリである。
メタデータとユーザー固有情報を分離するため、書誌原本を book_master、ユーザーの所持/進捗/評価を 
user_book として管理する。これにより、同一 ISBN のメタデータ再利用、複数ユーザー対応、外部 API 再
取得がしやすくなる。
ISBN 入力時は、まず openBD から書誌情報を取得し、取得できない場合は Google Books API へフォール
バックする。さらに見つからない場合でも、手動登録でアプリ利用が止まらない設計とする。
2. スコープ
2.1 V1 対象機能
 ログイン/ログアウト、ユーザー別データ分離
 本一覧、検索、状態タブ、並び替え、ページネーション
 手動登録、ISBN 登録、編集、論理削除
 状態・評価・メモ・タグ管理
 openBD 連携 + Google Books fallback + キャッシュ
2.2 V1 対象外
 ネイティブモバイルアプリ（iOS/Android）
 SNS 共有、公開レビュー、推薦アルゴリズム
 電子書籍ストアとの購入/蔵書同期
 バーコードカメラ読み取り（将来拡張）
3. 技術アーキテクチャ
アーキテクチャは、React フロントエンド、Spring Boot バックエンド、PostgreSQL データベース、外部
書誌 API 群から成る。責務を以下のように分離する。
層 技術 責務
Frontend React + TypeScript 画面表示、フォーム、検索条件管理、
API 呼び出し、表示最適化
Backend API Spring Boot 認証、業務ロジック、ISBN 正規化、
キャッシュ、外部 API 統合、監査ログ
DB PostgreSQL ユーザー、書誌原本、個人蔵書、タグ、
キャッシュ、ログ管理
External openBD / Google Books / Rakuten(将来) ISBN ベースの書誌取得、書影/説明補
完、購入導線
3.1 設計上の要点
 book_master と user_book を分離し、書誌メタデータと個人利用情報を混在させない。
 一覧 API はページネーション前提。検索条件は query parameter で受ける。
 外部 API の生レスポンスは isbn_lookup_cache へ保持し、正規化済み JSON も別に保存する。
 本削除は物理削除ではなく deleted_at による論理削除を基本とする。

8. ISBN 連携設計
ISBN 連携は本アプリの使い勝手を大きく左右するため、外部 API の利用順序、タイムアウト、キャッ
シュ、失敗時のフォールバックを明確化する。
8.1 取得優先順位
 1) openBD: 日本語書籍を主対象に最優先。/get?isbn=... を利用。
 2) Google Books API: openBD 未取得時のフォールバック。q=isbn:... を利用。
 3) 手動入力: 外部 API で見つからない場合でも登録継続可能。
 4) Rakuten Books: 初期 MVP では必須ではない。購入リンクや販売情報が必要になった時点で追加。
8.2 失敗時の方針
 外部 API タイムアウトは 2〜3 秒程度で打ち切り、UI には『自動取得に失敗したため手入力してくださ
い』を表示。
 一次 API 障害時は二次 API へフォールバックするが、二次 API も失敗した場合は 424 / 404 を返し、
フォーム入力継続は可能にする。
 外部レスポンス差分に備え、アプリ内部では title / authors / publisher / description / thumbnail_url 
などの共通 DTO へ正規化する。
連携先 URL 用途 注意点
openBD https://openbd.jp/ 日本語書誌・書影が強い。/
get?isbn=... を提供。

11. 非機能要件
区分 方針
性能 一覧 API p95 < 500ms、詳細 API p95 < 300ms、ISBN 検索 p95 
< 3s（外部遅延除く内部処理）。
可用性 単一 AZ でも可。1 日 1 回バックアップ、RPO 24h、RTO 4h を
初期目標。
セキュリティ HTTPS 前提。パスワードは強ハッシュ。CSRF/XSS/SQL 
Injection 対策。監査ログを保持。
拡張性 book_master と user_book を分離し、将来の複数ユーザー・
共有書誌・推薦機能へ備える。
運用性 request_id と structured logging を採用。外部 API タイムアウ
ト・再試行・サーキットブレーカを考慮。
整合性 更新系はトランザクション管理。重複登録防止は DB 一意制約
も併用。
キャッシュ ISBN キャッシュ TTL の標準は 7 日。出版社都合の更新に備
え、手動再取得 API を提供。
